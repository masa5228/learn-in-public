# /post-article — 学びを発信して経歴に蓄積するスキル

転職・対外アピールのため、記事を「執筆 → Zennで公開 → 経歴リポジトリに題目とURLを蓄積」まで一気通貫で行う。

## 前提・定数

| 項目 | 値 |
|------|----|  
| 経歴リポジトリ | `masa5228/learn-in-public`（Public, `main`） |
| 発信媒体 | Zenn（GitHub連携。`articles/` への push で自動公開） |
| Zennユーザー | `muramasa0228` |
| 公開URL | `https://zenn.dev/muramasa0228/articles/<slug>` |
| 実績の正本 | `career/index.json` |
| 実績の表示 | `README.md` の `<!-- articles:start -->`〜`<!-- articles:end -->` |
| 記事の雛形 | `docs/article-template.md` |

## 役割分担（コンシェルジュ運用）

- **調査・情報収集** → `researcher` 係に委任
- **GitHub操作（push・コミット）** → GitHub MCP で直接実行（自動）
- **執筆・統括・報告** → コンシェルジュ本体

## 確認ゲート（2箇所のみ・それ以外はすべて自動）

### ★ゲート1：テーマ確認・ディスカッション
スキル起動直後に行う。テーマが明確でない場合や転職アピール観点での深掘りが必要な場合はここでディスカッションする。以下が固まったらゲートを抜ける。

- テーマ・伝えたいこと
- 対象読者
- 種別 `type`（`tech` / `idea`）
- トピック候補
- タイトル要件：「固有名詞（技術名）＋実行系動詞（使う/作る/自動化した）＋筆者の立場」が入るか確認

### ★ゲート2：記事内容の確認・修正依頼
執筆した下書き全文を提示し、ユーザーが内容を確認・修正依頼できる唯一のタイミング。
「OK」または修正指示を受け取ったら、その後はすべて自動で完走する。

## 手順

1. **★ゲート1（テーマ確認）**: テーマ・狙い・対象読者・type・トピック候補をヒアリングし、必要に応じてディスカッションして確定する。
2. **調査・構成（自動）**: `researcher` 係に委任し一次情報を確認、見出し構成案を作る。
3. **slug採番（自動）**: `YYYY-MM-DD-<topic-kebab>` 形式。半角英数・`-`・`_` のみ、12〜50文字。既存と重複する場合は末尾に連番。
4. **執筆（自動）**: `docs/article-template.md` を雛形に、Zenn front matter 付き Markdown を作成。この時点では `published: false`。

   ### 文体ルール（AI臭の回避）
   - 文末を3文以上同じにしない（です/ます/でしょう/かもしれません/体言止めを混在）
   - 文頭の「また」「さらに」「したがって」は原則削除し、改行で切れ目を示す
   - em dash（—）・過剰な太字・絵文字を使わない
   - 各セクションに筆者の一次情報を最低1つ（失敗談・検証数値・認識の変化before→after）
   - 冒頭3行：①読者の課題をカギカッコのセリフで提起 → ②この記事で解決できると宣言 → ③結論（何が分かるか）を先出し

4.5. **人間味チェック（自動）**: 執筆直後に下記パターンを検出し、削除ではなく自然な表現に書き換える。意味と段落数は保持する。
   - AI語彙の多用：「〜と言えるでしょう」「重要な役割を果たす」「最適化」「本質的な〜」「効率的に」
   - 否定の並列・三語法の不自然な多用・冗長な要約段落
   - em dash（—）・過剰な記号・「です・ます」単調・文頭接続詞の過多・不要な箇条書き乱用

5. **★ゲート2（記事確認）**: 下書き全文と「タイトル／slug／公開URL（予定）／topics」を提示する。ユーザーが「OK」または修正指示を出す。修正があれば反映して再提示し、OKが出たら次へ。
6. **公開（自動）**: front matter を `published: true` にし、`articles/<slug>.md` を `main` に push（＝Zenn 自動公開）。
7. **記録（自動）**: `career/index.json` の `articles[]` 先頭へエントリを追記し、`README.md` の実績テーブルを再生成してコミット。
8. **報告（自動）**: タイトル・公開URL・topics・累計実績件数を日本語で簡潔に報告する。

## index.json のエントリ形式

```json
{
  "id": "2026-06-20-mcp-pipeline",
  "title": "記事タイトル",
  "platform": "zenn",
  "url": "https://zenn.dev/muramasa0228/articles/2026-06-20-mcp-pipeline",
  "published_at": "2026-06-20",
  "topics": ["mcp", "claude"],
  "summary": "一文要約",
  "status": "published"
}
```

`articles[]` は新しい順（先頭が最新）で保持する。

## README 実績テーブルの再生成

`<!-- articles:start -->`〜`<!-- articles:end -->` の間を `career/index.json` から生成し直す。

```
| 日付 | タイトル | 媒体 | トピック |
|------|----------|------|----------|
| 2026-06-20 | [記事タイトル](URL) | Zenn | mcp, claude |
```

## トラブル時

- GitHub への書き込みが 403: MCPトークンに `Contents: Read and write` と対象リポへのアクセスがあるか確認。
- Zenn に反映されない: Zenn 側のリポジトリ連携設定、`published: true`、slug 規則（12〜50文字）を確認。
