「AIに書かせた記事、なんかAIっぽいな」

Zennへの発信を自動化した翌日、自分の記事を読み返してそう思った。
確認ゲートを通過しているのに、文体がどこか機械的で引っかかる。
この記事では、その問題をどう特定し、スキルに改善を加えたかを共有する。

## 背景：Zenn発信を自動化した

先日、Claude Code と MCP を組み合わせて、記事執筆からZenn公開、GitHubの経歴リポジトリへの実績蓄積までを一気通貫で行う `/post-article` スキルを作った。

仕組みはシンプルで、テーマを伝えると Claude が調査・執筆・push まで実行してくれる。確認ゲートはテーマ確認と記事レビューの2箇所だけに絞ってあり、それ以外はすべて自動で動く。

初回の記事を公開したとき、正直「うまくいった」と思っていた。

## 翌日、違和感に気づいた

一晩おいて自分の記事を読み返すと、引っかかる箇所がいくつかあった。

- 「重要な役割を果たします」
- 「これにより、効率的に〜できます」
- 文末が「〜です。〜ます。〜です。」と3文続く

完全に間違ってはいないが、自分が書くときとは違う。こういう文体が積み重なると、読んでいる人には「AIが書いたな」とわかってしまう。エンジニアリングの文章として信頼を損ねる前に直したかった。

## 何が「AIっぽさ」を生むのか調査した

公開されているスキルやプロンプト集を調べたところ、英語圏では `blader/humanizer`（Claude Code スキル）などが体系化されていた。日本語技術ブログの分析も含めて、問題のパターンが見えてきた。

**構造の予測可能性がバレる原因**

語彙の問題というよりも、文章の構造がAI的になることが問題らしい。具体的には：

- 文末が単調に繰り返される（です/ます/ます/です）
- 「また」「さらに」「したがって」が文頭に来すぎる
- 一次情報がなく、一般論の羅列になる
- em dash（—）や過剰な太字が使われる（英語記事でよく見られる）
- 冒頭から結論まで、読者の共感を引く仕掛けがない

日本語特有の問題として、接続詞の過多と文末の単調さは英語向けスキルではカバーされていなかった。日本語版のルールを自分で作る必要があった。

## SKILL.md に3点を追加した

### 1. ゲート1（テーマ確認）にタイトル要件を追加

「固有名詞（技術名）＋実行系動詞（使う/作る/自動化した）＋筆者の立場」の構成が、Zennで読まれやすいタイトルの共通パターンとわかった。テーマを固める段階で意識するよう、確認項目に加えた。

### 2. 手順4（執筆）に文体ルールを明文化

```
- 文末を3文以上同じにしない（です/ます/でしょう/体言止めを混在）
- 文頭の「また」「さらに」「したがって」は原則削除、改行で切れ目を示す
- em dash（—）・過剰な太字・絵文字を使わない
- 各セクションに筆者の一次情報を最低1つ（失敗談・数値・認識の変化）
- 冒頭3行：①課題提起 → ②解決宣言 → ③結論先出し
```

Claude に指示を渡す段階でルールが明文化されていると、出力が変わる。曖昧な「人間らしく書いて」より、具体的な制約のほうが効く。

### 3. 手順4.5（校正ステップ）を新設

執筆直後、確認ゲートの前に自動で走るチェックを追加した。AI語彙クラスタや構造パターンを検出し、削除ではなく自然な表現に書き換える。意味と段落数は保持するというのがポイントで、単純な除去ではなく「書き直し」として指示している。

## やってみてわかったこと

スキルは一度作ったら終わりではない。

初版を実際に動かしてみると、想定していなかった品質の問題が出てきた。その問題を特定し、調査し、スキルに反映する。この改善サイクルを自分で回すこと自体が、ツールを使いこなすということだと思う。

「AIが書いた記事がAIっぽい」という問題は、一見些細に見える。ただ、継続的に発信して転職アピールや信頼構築につなげたいなら、文体の質は無視できない。読んだ人が「これは機械が書いたな」と思った瞬間に、伝えたいことの説得力が落ちる。

今後は改善後の記事を何本か出して、読み返したときの違和感がなくなっているかを確認していく。スキルの評価は、出力を見てからでないとわからない。

## まとめ

- `/post-article` スキルにAI臭の問題があることに気づいた
- 公開スキルと日本語ブログの調査から、原因パターンを特定した
- SKILL.md にタイトル要件・文体ルール・校正ステップの3点を追加した
- スキルは動かしてみないとわからない部分がある。改善サイクルを自分で回すことがツールの使いこなしにつながる
