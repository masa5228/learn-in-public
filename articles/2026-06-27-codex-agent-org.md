「AIに実装を任せると速いけれど、設計の一貫性が崩れていく」「かといって一番賢いモデルに全部やらせると、コストもクォータもすぐ底をつく」。AIコーディングを日常的に使う人なら、一度はこの板挟みに当たるはずだ。

この記事は、その板挟みを「1体の万能エージェント」ではなく「役割を分けた組織」で解いた記録になる。結論を先に置く。設計判断はOpus、重い実装はCodexに分け、その間に推論しない受付係を1枚挟む、という三層構成にした。なぜその線引きにしたのか、設計の意図を残しておきたい。

読み終えると、単一エージェントを強くするのとは別の方向、つまり「役割を分けて文脈を切る」という選択肢の輪郭が見えるはずだ。

## 何を作ったか

自分のローカル環境（Claude Code）に、開発専用の三層構成を足した。図にするとこうなる。

```
ユーザー
  │
  ▼
concierge（受付・橋渡し / Sonnet）   ← 推論しない
  │  開発依頼だけを architect に渡す
  ▼
architect（設計司令塔 / Opus）       ← 頭脳：基本設計・設計書・レビュー
  │  設計書 = Codex への契約
  ▼
Codex（実装エンジン / OpenAI・Plus枠）← 手足：詳細設計・実装・テスト
```

ポイントは、賢いモデルを1体置いて全部やらせるのではなく、「考える人」「手を動かす人」「取り次ぐ人」を分けたところにある。それぞれが見る文脈も、使うモデルも違う。

## なぜ「1体」ではなく「組織」にしたのか

単一エージェントに設計から実装まで通しでやらせると、会話が長くなるほど文脈が膨らむ。設計の議論、試行錯誤の実装ログ、テストの出力が全部1つの文脈に積み上がり、後半は判断がぶれやすくなる。

組織にする狙いは、賢さを足すことではない。文脈を切ることだ。設計は設計の文脈で、実装は実装の文脈で進めれば、それぞれが見るべきものだけを見られる。役割を分けるとは、責任を分けることであり、同時に文脈を分けることでもある。この発想が今回の構成の土台になっている。

## 役割分担①：設計はOpus、実装はCodex

一番伝えたいのはこの線引きだ。頭脳と手足を、別のモデルに割り当てた。

architect（Opus）がやるのは基本設計と、その成果である設計書づくり、それからレビューまで。設計書はプロジェクトの `docs/design/` に Markdown で置く。これを「Codex への契約」と位置づけた。曖昧さを残さず、後続の実装はすべてこの設計書に従わせる。設計書のパスを渡して、こう指示する。

```bash
codex exec --sandbox workspace-write \
  -C "<worktree の絶対パス>" \
  "docs/design/<機能名>.md に従って詳細設計・実装・テストを行え"
```

なぜここで線を引いたのか。理由は2つある。

1つは一貫性。設計という一番ぶれてはいけない判断を、長い実装ログで汚れていないOpusの文脈に閉じ込めたかった。設計の頭と実装の手を物理的に別プロセスへ分けると、設計の意図が実装の都合で薄まりにくい。

もう1つは現実的な制約だ。実装エンジンに使うCodex（Plus枠）は5時間のローリングウィンドウでクォータが決まっていて、常用やCI連携には向かない。だから「設計と軽いレビューはOpusが自分でやり、Codexへの委任は重い実装とテスト実装に絞る」というルールにした。クォータが切れたら、急ぎはarchitect自身が手を動かす。賢いモデルを贅沢に使い切るのではなく、ここぞという重い実装にだけCodexを差し向ける。分業の線は、品質と資源の両方から引いている。

## 役割分担②：間に「推論しない受付係」を置く

組織のもう1つの肝は、ユーザーとarchitectの間に concierge という受付係を挟んだことだ。そして concierge には、開発タスクで設計やタスク分解をさせない。

一見すると無駄な中継に見える。なぜ推論しない係をわざわざ置くのか。

ここでも理由は文脈の分離にある。concierge は依頼の受付、振り分け、結果の集約と日本語での報告に徹する。開発の長い試行錯誤はarchitectとCodexの側に閉じ込め、ユーザーが見る会話には「依頼」と「完了報告」と「判断が必要な上申」だけが流れるようにした。受付係が下手に推論を始めると、その判断がユーザー向けの文脈に混ざってしまう。だから「取り次ぐだけ」に役割を固定している。

もう1つ、受付役を置くときに見落としやすい落とし穴がある。concierge は全リクエストの通り道になる。ここに重い推論まで担わせると、相対的に非力なモデル（Sonnet）がそのまま組織全体の処理上限になってしまう。橋渡し役が天井になる構造だ。この「オーケストレーターのモデルがボトルネックになる問題」は、以前べつの記事で一度掘り下げた（[Claude Code でOpusを使い切る——設計はSonnet、実装はプロジェクト直起動](https://zenn.dev/muramasa0228/articles/2026-06-22-claude-code-opus-pattern)）。今回の三層構成は、その学びを土台にしている。対処はシンプルだ。受付役には推論をさせない。重い判断は必ず上の層（Opus）へ逃がし、concierge は受付・振り分け・取り次ぎだけに徹する。安い通り道としては使うが、天井にはしない、という割り切りである。

受付・頭脳・手足。三層に分けたのは、それぞれの文脈を独立させ、層をまたぐときに必ず要約（設計書・完了報告）を通すためだ。要約を挟むことが、文脈が無秩序に膨らむのを防ぐ関所になっている。

## 副次的な設計：安全と隔離

組織の骨格からは少し外れるが、実装を外部エンジンに委ねる以上、避けて通れない設計判断が2つあった。記録として残しておく。

1つはworktreeの隔離。Codexは必ず `git worktree` で切った作業ツリーの中だけで動かし、本体の作業ツリーは絶対に汚さない。実行は無人前提なので、サンドボックスはファイル編集だけ許してネットワークとシステムコマンドは塞ぐ設定にしている。生成された差分は本体から隔離された場所に溜まり、レビューを通ってから初めて本体へ反映される。

もう1つは課金事故の防止。Codexの実行エンジンはChatGPT（Plus枠）に固定し、APIキー経由の従量課金へ流れないよう多層で防いでいる。設定ファイルでログイン方法を `chatgpt` に強制し、実行のたびに環境変数のAPIキーを除去し、実行前に複数のチェックを通す。プロジェクトの `.env` に紛れ込んだAPIキーがログイン中でも黙って従量課金を優先してしまう挙動を踏んだことがあり、それ以来「キーが存在しないこと」を毎回手続きで確認している。$0の上限設定はもう使えないので、防御は設定と手順の側で多層に積むしかない。

この2つは主役ではない。ただ「手足を外注する」と決めた瞬間に、隔離と課金の設計が必ず付いてくる、という事実は書き留めておく価値がある。

## 作ってみての手応え

before/afterで言うと、以前は1体のエージェントに設計から実装まで通しで頼んでいた。速い代わりに、会話が伸びるほど設計の初志が薄れ、何を決めたのか自分でも追いにくくなっていた。

afterの今は、設計書という成果物がOpusの側に必ず残る。Codexはその契約に従って手を動かし、結果は隔離された場所でレビューを待つ。ユーザーの会話には依頼と報告だけが流れる。賢さを1点に足したのではなく、役割と文脈を分けたことで、判断の筋が通りやすくなった手応えがある。

もちろん万能ではない。層を増やした分だけ取り次ぎのコストは増えるし、Plus枠のクォータ管理という新しい面倒も抱え込んだ。それでも「1体を強くする」以外に「役割を分けて文脈を切る」という設計の引き出しを持てたのは、自分にとって大きい。

AIエージェントを組織として設計する人の、ひとつの実例として参考になれば嬉しい。
