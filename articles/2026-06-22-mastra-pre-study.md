## はじめに

「LLMをプロダクトに組み込みたいけど、どのフレームワークを選べばいいのか分からない」——そう感じたことはないだろうか。

この記事では、TypeScript製AIエージェントフレームワーク **Mastra** について調べた内容を整理する。**まだ実際には触っていない**。近く自分のプロダクト（EchoMask）への実戦投入を検討しており、その前に概念を頭に入れておくための事前学習メモだ。

結論を先出しすると、Mastraは「LLMに対話的に頼む道具」ではなく、**「LLMをコードの一部として組み込む枠組み」**だった。この違いを理解するだけで、どこで使うべきかが見えてくる。

## Mastraとは何か

[Mastra](https://mastra.ai/) はTypeScriptネイティブのOSSエージェントフレームワーク。Gatsby（静的サイトジェネレーター）を作ったチームが開発しており、Y Combinator W25バッチ出身で$13M調達済み。

- ライセンス：Apache 2.0（OSS）
- バージョン：v1.0 GA（2026年1月）、現在 v1.45.0
- GitHub Stars：25k超、npm週次DL：30万+

「TypeScript製」という点が重要で、PythonチームにはLangChainやCrewAIがあるが、TypeScript/Node.jsで同等のことをやろうとすると選択肢が限られていた。Mastraはその空白を埋める存在として設計されている。

## 「汎用AIアシスタント」との違い

調べていて最初に整理が必要だったのがここだった。

私はClaude Codeで複数のカスタムエージェントを動かしているが、あれは「AIに頼んで作業してもらう」使い方だ。Mastraは違う。

```
Claude Code（対話的）:
  あなた → Claude（脳）が判断 → ツールを呼ぶ

Mastra（組み込み型）:
  イベント/API → Mastraアプリ → LLMを部品として呼ぶ → ツールを呼ぶ
```

Mastraにおいて、LLMは「司令塔」ではなく「パーツのひとつ」。ロジックはTypeScriptコードで明示され、LLMはその中の特定のステップを担う。

一言でいうと：**「if-else/CASE文で書いていた条件分岐をLLMに置き換えて動的にする枠組み」**だと理解した。

静的なコードで列挙しきれない「曖昧・複雑・自然言語が絡む判断」をLLMに任せ、それ以外の制御フローはコードが担う。

## 主要機能の概観

### Agent

LLM + Toolの組み合わせでタスクを処理する。40以上のモデルプロバイダ（Anthropic/OpenAI/Gemini等）を統一APIで扱えるので、Claude→GPT-4oへの切り替えがコード変更最小で済む。

### Workflow

複数ステップをつないだグラフ型オーケストレーション。`.then()` / `.branch()` / `.parallel()` でフローを組む。

特徴的なのは **human-in-the-loop** が組み込みでサポートされていること。「このステップで一時停止して人間の承認を待つ」という処理をコードで書ける。承認が得られたら再開する、という構造だ。

### Memory

4層で構成されている：

| 層 | 内容 |
|----|------|
| 会話履歴 | スレッド単位のメッセージ保持 |
| Semantic Recall | 過去メッセージへの意味検索 |
| Working Memory | セッション間で保持する構造化データ |
| 自動圧縮 | 30,000トークンで自動圧縮（手動設定不要） |

自動圧縮が「手動設定不要」という点は地味に助かりそうだ。LangChainでは自分でメモリ戦略を選ぶ必要があり、そこで詰まった経験がある。

### MCP（双方向）

ここが自分の文脈で一番刺さった。

- **MCPクライアント**：外部のMCPサーバー（filesystemやplaywrightなど）をMastraエージェントのツールとして使える
- **MCPサーバー**：逆に、Mastraで作ったエージェント・ワークフローをMCPサーバーとして外部公開できる

つまり、MastraアプリをClaude Code / Cursor / VS Codeからツールとして呼び出せる。自分で運用しているconciergeエージェント群をMastraでラップしてMCP公開するユースケースが考えられる。

## なぜLangChainでなくMastraか

| 観点 | Mastra | LangChain/LangGraph |
|------|--------|---------------------|
| 言語 | TypeScript専用 | Python主体（JS版は遅延あり） |
| Serverless | ネイティブ対応 | 非対応（設計上） |
| DX | 外部ベンチマーク9/10 | 5/10 |
| MCP対応 | サーバー・クライアント双方 | 部分的 |
| エコシステム | 成長中（25k stars） | 圧倒的（1000+ integrations） |
| トークン消費 | 1.5〜2.5倍多め | 低消費 |

TypeScriptで書きたい、Serverlessに乗せたい、という条件が重なるとMastraの優位は明確だ。LangChainのエコシステムの広さは魅力だが、TypeScript環境で使おうとすると摩擦が大きかった経験がある。

## 自分の文脈でのユースケース

具体的にどこで使えるかを整理した（実戦前の想定）。

**EchoMaskの音声キャリブレーションシステム**が一番切実なユースケースだ。ユーザー登録時に肉声を録音し、LLMがボイスチェンジの設定値を生成する機能を開発凍結していた。当初はサーバーサイドからLLMに直接投げていたが精度が出なかった。

今思えば構造の問題で、音声解析（ピッチ・フォルマント抽出）とLLMによる設定値推定を分離できていなかった。Mastraで組み直すなら：

```
音声録音
  → 音声解析ツール（数値特徴量を抽出）
  → LLM（「この声質にぴったりな設定値は？」）
  → バリデーション（値が有効範囲内か）
  → 返却
```

このように「LLMに任せる箱」を明確に区切ることで、精度改善の糸口になると見ている。

他にはZenn記事生成パイプラインのWorkflow化や、GitHub PR自動レビューエージェントも候補だ。

## 注意点

調べた中で気になったネガティブ情報も記録しておく。

- **トークン消費が多め**：LangChainと比較したベンチマークで1.5〜2.5倍という実測値がある。コスト最適化が重要な用途では要注意
- **認証機能が実験段階**：`experimental_auth` はまだ本番品質ではない
- **日本語情報が少ない**：英語ドキュメントは整備されているが、日本語の事例記事はまだ少ない

## まとめ

Mastraを調べて得た最大の気づきは、**「LLMを使う」の意味が2種類あること**だった。

- 対話的に頼む（Claude Code等）
- コードに組み込む（Mastra等のフレームワーク）

後者のための枠組みがMastraで、TypeScript環境・Serverless・MCP双方向対応という条件が重なる場合の有力な選択肢になる。
