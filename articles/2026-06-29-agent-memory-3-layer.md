「会話が長くなると、少し前に決めたことをエージェントが忘れる」。LLMエージェントを継続して使うと、ほぼ全員がこの壁に当たる。
この記事は、その忘却を場当たりで殴らず、メモリを3層に分けて整理する"考え方"を共有するもの。
先に結論を書く。長期記憶は「賢い検索DBを建てる」より前に、ホット／索引／コールドの3層へ役割を分け、アクセスパターンから保存方式を逆算するのが効いた。派手な実装より、この設計順序のほうが効く。

## 出発点：作業記憶は有限である
コンテキストウィンドウは、エージェントにとっての作業記憶（RAM）に近い。容量は決まっていて、長い会話や複数セッションをまたぐと溢れる。溢れた瞬間に「さっき決めた方針」が文脈から抜け落ちる。

私はClaude Codeで複数エージェントの「コンシェルジュ構成」を運用している。セッションを切るたびに経緯が飛ぶのが地味に痛い。そこで長期記憶の設計を、既存ツールを触りながら考え直すことにした。

## 3層構造：hot / index / cold
たどり着いた骨格はシンプルだった。

- **Hot（作業記憶）**：常にコンテキストに載る。最小限に保つ。ここのサイズが毎回のトークンコストを決める。
- **Index（索引）**：何がどこにあるかを示す軽量マップ。常時ロードされるが、参照するのは「掘るか判断する瞬間」だけ。
- **Cold（ナレッジDB）**：大量の永続ナレッジ。普段はロードせず、必要なときに検索でだけ引く。

肝は「常時コストを hot + index に有界化し、cold は引くまでゼロにする」点にある。容量がいくら増えても、引かなければ毎ターンのコストには乗らない。

## 既存システムはどう解いているか
自分で結論を出す前に、長期記憶の仕組みを持つ実装をいくつか触って読んだ。驚いたのは、表現は違っても骨格がほぼ同じだったこと。

**MemGPT / Letta** はOSの仮想記憶になぞらえる。コンテキストをRAM、外部ストアをディスクと見立て、`main context`（作業記憶）／`recall`（過去メッセージ）／`archival`（ベクトルDBの長期保管）に分け、エージェント自身がツールコールでページングする。記憶を何に残し何を要約するかをエージェントが自律判断する「self-editing memory」が特徴だ（出典: MemGPT論文 arXiv:2310.08560、Letta公式ブログ）。

**HermesAgent**（Nous Research）は、`MEMORY.md` と `USER.md` によるファイル永続のホット記憶に加え、外部メモリプロバイダ（Mem0 / Hindsight など）をプラグインで1つ有効化し、各ターン前にプリフェッチしてシステムプロンプトへ注入する。長期保持は外部プロバイダ側のベクトル検索やナレッジグラフに委ねる構成だった（出典: Hermes Agent 公式docs）。

ほかにも **Mem0**（ベクトル＋グラフのハイブリッドで事実をADD/UPDATE/DELETE管理）、**Zep**（エピソード記憶と意味記憶を分けた時間的ナレッジグラフ）、**LangMem**（Semantic / Episodic / Procedural の3タイプ）と、設計思想はそれぞれ違う。

そして **Claude Code** 自体も、人手管理の `CLAUDE.md` と自動学習の Auto memory（`MEMORY.md` を索引に詳細はオンデマンド）に加え、コンテキスト使用が上限の約95%でサーバーサイド自動要約する compaction を持つ（出典: Claude Code / Anthropic Platform 公式docs）。

触る前と後で認識が変わった。before は「ツールごとに別物だろう」。after は「どれも hot / index / cold の3層に対応づく」。3層は、各実装を読み解くための共通言語として使える。

| 層 | MemGPT/Letta | HermesAgent | Claude Code |
|----|--------------|-------------|-------------|
| Hot | main context | MEMORY.md / USER.md | CLAUDE.md＋MEMORY.md索引 |
| Index | recall storage | provider prefetch | MEMORY.md索引 |
| Cold | archival(ベクトルDB) | 外部プロバイダ | 組み込みなし(MCP拡張) |

## 自分で建てて、そして捨てたもの
コールド層に、ナレッジグラフ型の `mcp__memory`（@modelcontextprotocol/server-memory）を使うつもりでいた。グラフDBがあるなら意味検索でリッチにできる、と思い込んでいたからだ。

実体を確かめて、その思い込みは崩れた。ソースを読むと検索関数には "Very basic search" とコメントがあり、中身は大文字小文字を無視した部分文字列マッチだけ。意味検索でも埋め込みでもない。さらに全操作がファイルの全件ロードで、`entityType` で検索を絞る手段もないため名前空間の分離もできない。

before は「グラフDBだから高機能」。after は「実体はキーワード検索＋全ロードで、`grep` ＋タグ運用と大差ないどころか、スケールで劣る」。ここで方針を変えた。

コールド層は素のMarkdownファイル＋`grep` にした。理由は単純で、`grep` はマッチした行しか返さない。だからアーカイブがどれだけ育っても、引くまでは毎セッションのコストに乗らない。意味検索が無い弱さは、各ファイルに別名や同義語タグを厚く焼いて字句一致の再現率を底上げすることで埋める。

副産物の気づきもあった。記憶には「蓄積型（事実・決定）」と「状態型（プロジェクト進捗・Todo）」があり、更新規律が逆だ。前者は積み、後者はその場で上書きする。これを混ぜると、古いスナップショットが残って「どれが今正しいのか」が分からなくなる。状態は積むものではなく、常に最新の1版に保つものだった。

## 検索方式：grep か、ベクトルか
最後に残るのが検索の選択だ。一般論として、ベクトル検索は意味的な近傍を広く拾える反面、コストと運用が重い。字句検索（grep / BM25）は完全一致や識別子に強く、インフラなしで即動く。

数字でも裏が取れる。BM25とベクトルをRRFで統合しクロスエンコーダでリランクするハイブリッドが Recall@5 で 0.816 に達する一方、単独のBM25は 0.644、Dense は 0.587 にとどまる（出典: 各種RAG検索ベンチ）。ハイブリッドは強い。ただしそれは相応の構築コストと引き換えだ。

個人やチーム規模のエージェントなら、最初は grep で十分に回る。ナレッジが数百件を超えたあたりで、コールド層だけハイブリッドへ差し替えればいい。基盤の豪華さではなく、アクセスパターンに合った最小構成から始めるのが費用対効果に合う。

## まとめ
長期記憶の設計で効いたのは、技術選定そのものより順序だった。

1. まず役割を hot / index / cold の3層に分ける
2. cold は遅延ロードにして、常時コストを hot + index に閉じ込める
3. 検索方式は「あるから使う」ではなく、アクセスパターンと規模から逆算する

MemGPT / Letta も HermesAgent も Mem0 も、骨格はこの3層に落ちる。だからこそ、自分のケースでは「グラフDBがあるから使う」ではなく「実体を確かめてから選ぶ」が分かれ目になった。確かめた結果、いちばん地味なファイル＋grepが、いちばん要件に合っていた。

派手な記憶基盤を建てる前に、まず3層で役割を切る。これが、既存ツールを一通り触って残った結論だ。
