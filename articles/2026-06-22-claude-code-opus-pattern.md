## はじめに

「Opusをサブエージェントとして呼べばいいか」——そう思って構成を考えたら、思ったより性能が出ない構造になることに気づいた。

この記事では、Claude Codeのマルチエージェント構成でOpusの性能を引き出せない理由と、設計と実装を別セッション・別モデルに分離して解決するパターンを紹介する。

結論から言うと、**オーケストレーターと実装者のモデルを一致させる**ことが鍵だった。

## オーケストレーターがボトルネックになる仕組み

Claude Codeでマルチエージェントを組むとき、こういう構成を考えやすい。

```
コンシェルジュ（Sonnet）
  ↓ Agent(model="opus") でOpusを呼ぶ
  Opusサブエージェント
    ↓ ファイル編集・実装
    ↓ 結果を返す
  コンシェルジュ（Sonnet）が結果を受け取る
```

「Opusに実装させた」ように見える。Opusがコードを書く部分はOpusだ。

問題は**その後**にある。

Opusが書いたコードの結果はコンシェルジュ（Sonnet）に戻ってくる。複数ファイルにまたがる変更、ビルドエラーへの対処、実装方針の微調整——これらの判断はすべてコンシェルジュが担う。サブエージェントはメインセッションの文脈やファイル状態を引き継がないので、Opusは毎回ゼロから状況を把握する必要もある。

結果として、実装における重要な判断がSonnetに戻ってくる。Opusを呼んでも、オーケストレーターの性能が実質的な上限になる。

## 解決策：プロジェクトディレクトリでOpusを直接起動する

Claude Codeはプロジェクトディレクトリに `.claude/settings.json` を置くことで、そのディレクトリ専用のモデルを指定できる。

```json
{
  "model": "claude-opus-4-8"
}
```

このディレクトリで `claude` を起動すると、Opusがオーケストレーターかつ実装者になる。

```
プロジェクト/でclaudeを起動（Opus）
  → Opus自身が判断・実装・確認・修正をすべて担う
  → 文脈の断絶なし
  → 性能がそのまま出る
```

サブエージェントとして「使われる」Opusではなく、セッションの主体としてのOpusになる。

## CLAUDE.mdを引き継ぎ書にする

設計はSonnetで十分だ。コンシェルジュセッションで仕様を詰め、決定事項をCLAUDE.mdに書き残せば、次にOpusが起動したとき最初に読み込む。

```
Sonnetセッション（コンシェルジュ）:
  設計・仕様決め
    → 決定事項をCLAUDE.mdに記録

Opusセッション（プロジェクト直起動）:
  CLAUDE.mdを読む
    → 迷わず実装・CI/CD
```

CLAUDE.mdが引き継ぎ書として機能する。セッションをまたいで文脈が共有されなくても、ファイルに書いてあるので問題ない。

## EchoMaskでの実践

自分が開発している匿名通話アプリ「EchoMask」に凍結中の機能があった。ユーザーの肉声を録音してボイスチェンジ設定値を生成するキャリブレーション機能だ。

Sonnetセッションで設計を詰めた。

- 廃止の原因特定（Geminiクォータ切れ時にランダムプリセット生成していた）
- 再実装フローの設計（音声録音→Gemini分析→フロントで品質スコア算出→ユーザー確認→保存）
- フォールバック廃止・バリデーション追加の方針

これをCLAUDE.mdに書き起こし、`.claude/settings.json` でOpusを指定した。次にプロジェクトディレクトリで `claude` を起動すれば、Opusが設計書を読んで実装に入る。

設計のコストをSonnetで払い、実装の品質をOpusに担わせる構造になった。

## 役割分担の整理

| セッション | モデル | 担当 |
|-----------|--------|------|
| コンシェルジュ/汎用 | Sonnet | 調査・設計・仕様決め・CLAUDE.md記録 |
| プロジェクト直起動 | Opus | 実装・テスト・CI/CD |

設計と実装は性質が違う。設計は会話ベースで広いコンテキストが活きる。実装は文脈を絞って深く掘り下げる方が良い。モデルもセッションも分けることで、それぞれの強みが出る。

## まとめ

Claude Codeで「Opusをサブエージェントとして呼べば高性能になる」は半分しか正しくない。重要な判断がオーケストレーターに戻る限り、オーケストレーターのモデルが実質的な上限になる。

解決策は `.claude/settings.json` にOpusを指定してプロジェクト直起動。CLAUDE.mdを設計の引き継ぎ書として使えば、Sonnetでの設計とOpusでの実装を自然につなげられる。

コストと性能の両方を取れる構成だと思っている。
