## はじめに

「コメントは書かない方がいい」——そう言われたとき、正直リーダブル原理主義的な過激思想に聞こえた。

この記事では、なぜコメント禁止規約をプロジェクトに採用したのかを説明する。理由はClean Codeの教えへの信仰ではなく、**AIコーディングツールがコメントに騙されるから**だ。

結論を先に言うと、2024年以降の開発において「コメントは人間の可読性を上げる」という前提は崩れはじめており、「AIのミスを減らす」という新しい基準で規約を組み直す合理性がある。

## コメント禁止の古典的な理由

2008年、Robert C. Martin（通称 Uncle Bob）が『Clean Code』でこう主張した。

「コメントはコードで意図を表現できなかった失敗の証拠だ」

根拠はシンプルだった。コードは更新されるがコメントは更新されない。時間が経つほどコードとコメントの乖離が広がり、コメントは「腐る」。腐ったコメントはないよりも有害だ、という論理だ。

解決策は自己文書化コード。変数名と関数名を丁寧につければ、コメントなしでもコードが読めるという考え方だ。

```typescript
// 悪い例：コメントに頼る
// 週の開始日（月曜日）を取得する
const d = getMonday(new Date())

// 良い例：命名で読める
const weekStart = getMondayOfCurrentWeek(new Date())
```

これ自体は今も有効な考え方だ。ただ、日本ではこの規約が広く馴染まなかった背景があると思っている。

ひとつの仮説として、**コードが英語前提である**という問題がある。識別子・キーワード・ライブラリのAPIはすべて英語だ。ネイティブの英語話者にとって「コードを読む」のは自然言語に近い体験だが、日本語話者にとっては異言語を読む作業に近い。コメントが日本語で書かれていれば「翻訳レイヤー」として機能し、英語コードの読解コストを下げてくれる。コメントを禁じるということは、その補助輪を取り外すことを意味した。Clean Codeの主張が「英語ネイティブの開発文化から生まれた前提」を持っていたとすれば、日本での抵抗感は理解できる。

## AIコーディングで起きる問題

LINE WORKSのBotを設計しているとき、こういう場面を想定した。

```typescript
// step=0のときはメッセージを無視する
if (session.step === 0) {
  return sendFixedMessage(userId)  // 仕様変更で固定メッセージ送信に変えた
}
```

コメントは「無視する」と書いてある。コードは「固定メッセージを送る」。仕様変更でコードだけ更新して、コメントが古いまま残った状態だ。

人間の開発者ならこう判断する。「コメントより新しいコードの方が正しいはず」。経験則で補正できる。

OpusにこのBotの挙動修正を依頼するとどうなるか。LLMは自然言語であるコメントを高い信頼度で処理する。「コメントには"無視する"と書いてある。コードが矛盾している。これはバグだ」と解釈して、`sendFixedMessage()`を削除し、無視する実装に戻す可能性が高い。

**意図していた仕様を、AIが親切にバグ修正として壊す。**

## なぜ人間より深刻か

LLMの特性の問題だ。

LLMは自然言語の処理を得意とする。コメントは自然言語であり、コードより「意図」として読みやすい。コードとコメントが矛盾したとき、LLMはコメントを「ドキュメント」として優先する傾向がある。

人間は「コミット履歴」「ファイルの更新日時」「周辺コードの文脈」を総合して判断できる。LLMはコンテキストウィンドウ内の情報で判断する。腐ったコメントが視野に入ると、それが正として機能してしまう。

コメントを書いた時点で、それは意図の宣言になる。AIにとっては特に。

## 現代では人間よりAIへの配慮が優先される

AIコーディングが普及した結果、コードを深く読む主体が変わりつつある。

機能追加・バグ修正・リファクタリングの多くをAIが担う場面では、コードは「人間が読むドキュメント」である前に「AIが処理するインプット」だ。「コメントを書いて人間の可読性を上げる」より「コメントを書かずAIの誤読を防ぐ」方が、チームへのインパクトが大きい局面が増えている。

コーディング規約は「誰が読むか」を想定して作る。読む主体が変わった以上、規約の基準も変わる。

## まとめ

コメント禁止規約の根拠は2層になった。

- **古典的な理由**（2008〜）：コメントは腐る・自己文書化コードで十分
- **AI時代の理由**（2024〜）：腐ったコメントがLLMを混乱させ、意図しないバグ修正を引き起こす

「リーダブル原理主義」ではなく「AIのミスを減らす規約」として再定義すると、採用の合理性がクリアになる。

WHYが非自明な制約（APIの制限・回避策・意図的な設計判断）はコメントで残す。それ以外は書かない。AIが読んでも混乱しない、シンプルなコードベースが目標だ。
